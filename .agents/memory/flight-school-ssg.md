# Flight school static pre-rendering (SSG)

The flight-school artifact is a Vite CSR SPA that is pre-rendered to real static HTML at build time so every page is crawlable.

## PRODUCTION SERVING MODEL CHANGED: Express, not static host
- Production no longer uses `serve="static"`. The **Express api-server serves the prerendered marketing site**: root `.replit [deployment]` is `build = "pnpm run build"` (root build → builds flight-school + api-server) and `run = "pnpm --filter @workspace/api-server run start"`. `api-server/src/app.ts` resolves `publicDir = ../../flight-school/dist/public` (works because api-server is ESM `.mjs`, so `import.meta.url` is real), `express.static(publicDir, {extensions:["html"], redirect:false})`, then a fallback.
- **Real HTTP 301s now live in app.ts** (not possible on the old static host): trailing-slash `/p/` → `/p` and `/p.html` → `/p`. These are **scoped away from `/api`** (an `isApiPath()` guard runs first) — never 301 API routes (would break non-GET clients).
- **Hard 404 for unknown routes** (fixes soft-404 SEO issue): every valid marketing route is a prerendered flat file served by `express.static`; anything reaching the fallback is genuinely unknown → `res.status(404).sendFile(404.html)` (GET/HEAD only). `404.html` is prerendered from `render("/404")` → NotFound page (which sets `robots="noindex, follow"` via the new `Seo` `robots` prop).
- **artifact.toml `[[services.production.rewrites]]` are now LEGACY/dev-proxy only** — production canonicalization is done by Express, not rewrites. The flat-file prerender + rewrite notes below still describe how the static files are produced/served, but the production 301/404 behavior is Express's.

## Flat-file rewrite gotcha (static-host history; still informs flat-file prerender)
- Root cause confirmed against the LIVE site: when a route is prerendered as a DIRECTORY (`/route/index.html`), the Google static host treats `/route` as a directory and **301-redirects `/route` → `/route/`**. That 301 fires BEFORE Replit rewrites, so a `from="/route"` rewrite never matches. Then `/route/` is NOT "an existing file" (it's a dir), so shadowing does NOT apply and the catch-all `/* → /index.html` hijacks it → homepage HTML. Net: clean URLs served homepage; only `/route/index.html` worked. (Per-route `from="/route"` rewrites did NOT fix this — the 301 defeats them.)
- **FIX (current): prerender FLAT FILES — `dist/public/<route>.html`, NOT `<route>/index.html`.** With no directory, `/route` is neither a file nor a dir, so no 301; explicit `[[services.production.rewrites]] from="/<route>" to="/<route>.html"` serves the right page. Real assets (`/assets/*`) shadow rewrites and serve directly. SPA catch-all `/* → /index.html` stays LAST (handles non-prerendered alias routes + true 404s). Redirect stubs are also flat (`<old>.html`).
- Replit static rewrite semantics (from docs): rewrites are evaluated **in listed order**; a rewrite is ignored only when `from` matches an **existing file** (called *shadowing*) — directory-index resolution does NOT count. Config lives in `artifact.toml` `[[services.production.rewrites]]` (monorepo), edited only via `verifyAndReplaceArtifactToml`.
- **Generate rewrites from sitemap `<loc>` + `redirects.ts` keys, never hand-maintain.** ONE rewrite per path: `from="/<p>" to="/<p>.html"`, then catch-all `/* → /index.html` last. Current = 86 explicit + catch-all = 87.
- Currently 86 explicit clean-URL rewrites + catch-all = 87 (no trailing-slash variants). A historically-indexed `/<p>/` hits the SPA catch-all (homepage); acceptable because flat files never generate/301 to `/<p>/` and the sitemap + all internal links use clean `/<p>`, so Google re-consolidates.
- **NOTE: there is NO confirmed ~100 rewrite limit.** An earlier guess blamed the publish failure on rewrite count (173 vs 87) — that was WRONG. The real cause was the PORT gotcha below; the 173-rewrite build failed on PORT before rewrites ever mattered. Do not treat rewrite count as the deploy blocker.

## CRITICAL deploy gotcha: EVERY artifact's vite.config must NOT require PORT/BASE_PATH for builds
- The publish/deploy build runs `pnpm run build` at the repo root = `pnpm run typecheck && pnpm -r --if-present run build`, which builds/typechecks **every workspace package** — including dev-only artifacts like `mockup-sandbox` that are never actually deployed. The deploy build env does NOT set PORT or BASE_PATH. Any `vite.config.ts` that throws at config-load when they're missing → **publish FAILS** with `Error: PORT environment variable is required`, even though local `pnpm build` (with env set) passes. This bit BOTH flight-school AND later mockup-sandbox — fix it in **every** artifact's vite.config, not just the one you're deploying.
- Diagnose via `getDeploymentBuild({buildId})` logs (`listDeploymentBuilds` to find the failed id) — `fetchDeploymentLogs` shows nothing for failed builds. The failing line names the package: `artifacts/<name> build: Error: PORT environment variable is required`.
- **Fix:** only require/parse PORT when `command === "serve"` (dev/preview), and default `basePath = process.env.BASE_PATH ?? "/"`. **Overload pitfall:** flight-school uses `defineConfig(async ({command}) => {...})` (returns Promise) and that typechecks; but mockup-sandbox's same async form FAILED typecheck with TS2769 "No overload matches / has no properties in common with UserConfig" (TS wouldn't pick the Promise-returning overload there). Fix that one by keeping the config fn **synchronous** — hoist the dev-only `await import(cartographer)` to a module-top-level `const devPlugins` (top-level await is fine in ESM vite config) so the fn returns a plain `UserConfig`. Verify each: `env -u PORT -u BASE_PATH NODE_ENV=production pnpm run build` exits 0; dev workflows still serve 200.
- **Rewrites only apply in PRODUCTION (static deploy).** Dev workflow uses Vite, so you cannot verify rewrite behavior locally — must republish, then `curl -sS -D- https://<domain>/<clean-url>` (expect 200 + route-specific title/canonical, NOT 301-to-homepage).

## How it works
- `build` script chains three steps: client `vite build` → SSR build `vite build --ssr src/entry-server.tsx --outDir dist/server` → `node prerender.mjs`.
- `prerender.mjs` reads `dist/public/index.html` as a template (must contain `<!--app-head-->` and `<!--app-html-->` placeholders), renders each route, writes `dist/public/<route>.html` (FLAT files — see rewrite gotcha above; root stays `index.html`). Clean URLs need the explicit per-route rewrites above.
- `App` accepts optional `ssrPath` + `helmetContext` props (passed to wouter `Router ssrPath` and `HelmetProvider context`). `entry-server.tsx` renders App and extracts helmet head.

## Decisions / gotchas
- **MUST use `react-helmet-async@2.0.5`, NOT 3.0.0.** 3.0.0 is a React-19 fork that relies on React's native metadata hoisting (it ships a `React19Dispatcher`); with `renderToString` it leaves `helmetContext.helmet` EMPTY and dumps every SEO tag inline inside `<body><div id="root">` instead of `<head>`. View-Source then shows NO meta in `<head>` — the exact bug a user will report. Fix is 2.0.5 + the noExternal line below. **Don't "upgrade" helmet back to 3.x.**
- **2.0.5 ships CJS**, so its named-export ESM import breaks the SSR bundle unless Vite bundles it: `vite.config.ts` has `ssr: { noExternal: ["react-helmet-async"] }`. Keep it.
- **helmet 2.0.5 dedupes meta by name/property and dedupes identical `<link>`/title across separate `<Helmet>` instances**, but does NOT dedupe two canonicals with *different* hrefs. **Rule:** a given tag type must have exactly one source. The per-page `<Seo>` component (`src/components/Seo.tsx`) is the sole source of `<title>`, description, robots, canonical, og:* and twitter:* (all route-specific, derived from each page's `title`/`description` props). The global `SeoSchema` (in `Layout`) is now ONLY the Organization/LocalBusiness JSON-LD. Don't reintroduce overlap (FaqPage adds a 2nd JSON-LD of a *different* @type via `<Seo>` children — that's fine, distinct schemas).
- **og:title==page title, og:description==page description, twitter:title/description mirror them, canonical==og:url==route URL; twitter:image stays the default aircraft-exterior.png.** Every page renders `<Seo title=... description=.../>` (LandingPage/AirTourPage pass `page.metaTitle`/`page.metaDescription`). Do NOT put hardcoded homepage og/twitter values in a global component again — that was the SEO regression.
- **Prerender route list = `public/sitemap.xml` `<loc>` entries (canonical URLs)** PLUS an explicit `aliasRoutes` list in `prerender.mjs` for the client-only short routes (`/pricing`, `/contact`, `/cfi-training`, etc.). **These aliases MUST be prerendered** now that the Express server returns a HARD 404 for any non-prerendered path — if you drop an alias from `prerender.mjs` it will 404 in production. Keep `aliasRoutes` (prerender.mjs) in sync with `ALIAS_CANONICAL` (Seo.tsx); aliases canonicalize to their sitemap URL so they're not duplicate-content even though they 200. Internal `<Link>`s should still prefer canonical (sitemap) URLs.
- **`main.tsx` deliberately keeps `createRoot` (not `hydrateRoot`).** Avoids hydration-mismatch fragility across 85 pages given the helmet quirk; the prerendered HTML serves crawlers/first paint and the client re-renders identically. **Why:** correctness/robustness over the minor re-render cost.
- `index.html` holds only global non-duplicated tags (charset, viewport, gtag G-MF17HGD9C9, favicon, fonts) plus the two placeholders. All SEO title/meta come from helmet.
- Build/prerender does NOT need `PORT`/`BASE_PATH` (vite.config now defaults them — see deploy gotcha above). Dev workflow still provides them; `PORT` is only required for `command === "serve"`.

## URL consolidation & canonicals (now Express 301 + soft stubs)
- Production is now Express-served (see top section), so trailing-slash/.html canonicalization uses **real HTTP 301s** in app.ts. The `redirects.ts` soft-redirect stub mechanism is still used to retire/merge a *content* duplicate URL: drop it from `landingPages`/sitemap, add the old→canonical pair to `src/data/redirects.ts`, have `App` route it through a wouter `<Redirect replace>`, and have `prerender.mjs` emit a flat stub `<old>.html` with `rel=canonical` to the target + `<meta name="robots" content="noindex,follow">` + `<meta http-equiv="refresh">`. Build log shows `N pages, A aliases, M redirect stubs`.
- **Canonical + og:url are computed in `Seo.tsx` from wouter `useLocation`** → `canonicalUrl = SITE_URL + (location==="/" ? "/" : location)` with `SITE_URL="https://acceleratedflightschool.net"` (no trailing slash). `<Seo>` renders per-page inside the router so the hook resolves the current path (incl. SSR). An `ALIAS_CANONICAL` map in `Seo.tsx` maps the client-only alias routes (`/pricing`, `/contact`, `/cfi-training`, etc., NOT in sitemap) to their canonical sitemap URL so a direct hit still canonicalizes correctly. (This logic moved out of `SeoSchema.tsx`, which is now JSON-LD only.)
- Per-page unique prose lives in slug-keyed `Record<string,UniqueSection>` maps split across files (`landingUniqueAreas`/`landingUniqueTopics`/`airTourUnique`, merged in `landingUnique.ts`) so multiple subagents can populate them in parallel without clobbering the 1000+-line `landingPages.ts`. **Why:** concurrent edits to one big data file collide; separate files keyed by slug don't.
